Case Study: Optimization of a Data Processing Pipeline in a Financial Industry

  • Description of the Initial Pipeline

    At some point of the process, the entry point app sends a notification to a Messaging broker where other modules acting as Listeners can consume falling messages and execute tasks if any.

    The request contains metadata that defines the type of processing to be triggered and the data that is stored in one or multiple files. The Framework supports multiple file formats.

  • Challenges Faced

    In some processes, there were stored SQL procedures to execute sequentially, which was taking a lot of time.

    The messaging broker that was actually in use was outdated and not scaling, so a transition to the newest was a requirement.

    Legacy to Amazon Connect

    Smart interactive voice response(IVR)

  • Solutions Provided

    We initially explored different approaches to address the issue of long procedure execution times. We focused on two main options later.

    The first was about using the reactive programming approach, which relies on asynchronous event processing and data streams. We implemented a proof of concept (POC) using Spring Data R2DBC. R2DBC stands for Reactive Relational Database Connectivity, which is a specification for integrating SQL databases using reactive drivers.

    Legacy to Amazon Connect

    Smart interactive voice response(IVR)

    The second option was to use the Java async library with the Completable Future. This would allow us to execute multiple tasks simultaneously, and to treat exceptions if they occurred during computation.

    After considering the compatibility of the solution with an "aged" system and the current DB server, we decided to go for the second option. This involved integrating and deploying a solution that manages parallel execution of stored procedures. This resulted in a significant impact on the performance of the framework, which in turn affected the business.

    In regard to the transition to a new message broker that facilitates enhanced communication between our applications, two solutions are deployed. The first, which is primarily intended for use by developers, is an Active MQ broker. The second, which is employed in both test and production environments, is an Amazon SQS message broker. The advantage of this middleware is that it supports industry-standard protocols for queuing across languages and platforms.
